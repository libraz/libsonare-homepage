# True Peak 安全性

True Peak 安全性とは、マスター済みファイルが再構成、コーデック変換、配信再生の後にクリップしないよう、十分な余裕を残す考え方です。

サンプルピークは、デジタルファイルに保存された値だけを示します。再生機器やエンコーダーは、そのサンプル間を補間して連続した波形に戻します。この再構成された波形が、保存されたサンプルピークを超えることがあります。つまり、サンプル上は安全に見えるマスターでも、変換後に歪む可能性があります。

## なぜサンプルピークだけでは足りないのか

デジタル音声ファイルは滑らかな曲線ではなく、離散的なサンプル点の列です。DAC やエンコーダーが音に戻す際は、サンプル点を補間して連続波形を再構成します。この再構成波形は、保存された点そのものより高くなることがあります。

特に問題になりやすい場面は次のとおりです。

| 状況 | リスク |
|------|--------|
| 不可逆エンコード | MP3・AAC・Opus などへの変換で新しいピークが生まれることがある。 |
| 強いリミッティング | 密度の高いマスターほど、再構成時の飛び出しに弱い。 |
| ストリーミング側の正規化 | 配信プラットフォーム側で再エンコード／再処理を通る場合がある。 |
| 再生機器側 | 0 dBFS ぎりぎりの素材を苦手とする再生チェーンが存在する。 |

True Peak 計測では、信号をオーバーサンプリングしてからピークを調べることで、このリスクを推定します。

## 実用的なシーリング

一般的なリリース時のシーリングは `-1 dBTP` 前後です。コーデック変換に対する余裕を広く取りたい場合は `-2 dBTP` を使うこともあります。逆に、納品先が明確で管理された環境であれば、もう少しタイトなシーリングを採用することもあります。

シーリングはラウドネスのターゲット値ではなく、あくまで安全マージンです。静かなファイルでもピークが危険なことはありますし、大きいファイルでも True Peak の範囲内に収まっている例は珍しくありません。

## リミッターのルックアヘッド

True Peak リミッターは、ピークが到達する前に動き出す必要があります。ルックアヘッドはそのための先読み時間です。短すぎると速いピークを取り逃しやすく、長すぎるとレイテンシが増え、トランジェント感も変わってきます。

オフラインのマスタリングであれば、数 ms 程度のルックアヘッドは通常問題になりません。ライブ処理やインタラクティブ用途では、レイテンシの影響が大きくなる点に注意します。

## libsonare での扱い

libsonare のマスタリングチェーンには True Peak リミッターのステージがあり、Studio モードでシーリングとルックアヘッドを調整できます。WASM の ISP ベンチマークでは、ITU-R BS.1770-4 が True Peak 測定に求める最小要件である 4 倍オーバーサンプリングと、スライディングマックスのガードを組み合わせ、ブラウザ内レンダリングでも十分速くピーク検出できることを確認しています。

:::: details 実装メモ

安全確認の考え方は True Peak メータリングと同じです。オーバーサンプリングで再構成波形を近似し、密になった格子上で最大値を評価します。元のサンプル点だけをクランプするリミッターでは、サンプル間オーバーが残ってしまうため、ガードは入力バッファではなくオーバーサンプリング後の信号経路を監視する必要があります。

ルックアヘッドとリリースは独立に決められません。ルックアヘッドはピーク到達前に減衰を開始する時間を与え、リリースはユニティゲインに戻る速さを決めます。リリースが短すぎるとモジュレーション（音色の揺れ）が出やすく、長すぎるとトランジェントが鈍り、稼げるラウドネスヘッドルームも減ります。実用値は、コーデックセーフティ／トランジェント保持／聴感上のゲインムーブメントの折り合いで決まります。
