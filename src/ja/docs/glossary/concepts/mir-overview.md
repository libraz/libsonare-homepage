# MIR の全体像

MIR は Music Information Retrieval（音楽情報検索）の略です。テンポ、ビート位置、キー、コード、ピッチ、音色、セクションなど、音から音楽的な構造を取り出す音声解析の領域です。

ここで挙げる用語は単独の機能名ではありません。多くの MIR タスクは、スペクトル表現や時間-周波数表現を土台に構築されています。

## タイミング: BPM、ビート、オンセット、セクション

BPM はテンポを推定します。ビートトラッキングはタイムライン上に拍を置きます。オンセット検出はノート・ドラム・子音などのイベントが始まる瞬間を探します。

セクション解析は、イントロ・Aメロ・サビ・ブレイク・繰り返しパターンなど、長めのスパンをまとめて扱います。通常はリズム・ハーモニー・音色・エネルギーの時間変化に基づいて推定します。

## ハーモニー: キー、コード、クロマ

キー検出は楽曲の調性中心（トニック）を推定します。コード認識はその瞬間に鳴っているコードを推定します。クロマは周波数成分を 12 のピッチクラス（C・C#・D・…・B）に畳み込んだ表現で、同じピッチクラスは複数のオクターブにまたがって寄与するため、ハーモニー解析と相性が良いのが特徴です。

クロマはハーモニーには強力ですが、オクターブの違いや音色情報の一部を捨てています。コードやキー解析では利点ですが、他のタスクでは制約になります。

## スペクトル: FFT、STFT、スペクトログラム

FFT は DFT（離散フーリエ変換）の高速実装で、サンプルブロックを周波数成分に変換します。STFT はそれを短いウィンドウで繰り返し適用し、周波数成分を時間方向にも追えるようにします。

スペクトログラムはその可視化です。横軸が時間、縦軸が周波数、明るさや色が強度を表します。

## 知覚特徴量: メル、MFCC、CQT、VQT

メルスペクトログラムは、人間の聴覚に合わせて周波数分解能をひずませます。MFCC は log-mel パワースペクトルに DCT（離散コサイン変換）をかけて得られる係数で、スペクトル包絡を低次元で表現します。

CQT と VQT は、音楽的なピッチ間隔に合わせて周波数ビンを配置します。Hz 等間隔の FFT より、ピッチ関係そのものを扱いたい場面で有効です。

## 分離とピッチ: HPSS とピッチ推定

HPSS（Harmonic / Percussive Source Separation。倍音成分と打撃成分の分離）は、スペクトログラム上で水平方向に伸びる持続的な倍音成分と、垂直方向に立ち上がる打撃成分を分離します。ドラムとピッチ楽器は互いの解析を乱しやすいため、下流タスクの精度向上に役立ちます。

ピッチ推定は基本周波数を追跡します。メロディ、ボーカル、単音楽器、チューニング確認、採譜系のワークフローで使えます。

:::: details 実装メモ

libsonare はブラウザ（WASM）、JavaScript、Python、ネイティブバインディング、CLI、C++ API のいずれからも MIR 関連機能を呼び出せます。多くの機能は STFT・クロマ・スペクトルエネルギー曲線といった中間表現を共有しているため、同じ素材に対して BPM・キー・コード・セクションを連続で取り出すような使い方をしても、内部の重い計算が一度で済むよう設計されています。

ブラウザの解析デモは対話的な解析を主目的にしており、`StreamAnalyzer` 経由のプログレッシブな BPM／キー／コード推定や、AudioWorklet 連携を前提にしています。一方でマスタリングデモは、ラウドネス計測・リファレンス比較・レポート出力といった計測寄りの API に焦点を当てています。同じライブラリの上に位置付けが異なる 2 つのデモを載せることで、解析と仕上げの両面で何が再利用できるかを示しています。
