# 配信ターゲット

配信ターゲットは、公開先に合わせて決めるラウドネスとピーク安全性の設定です。

デモでは Spotify、YouTube、Apple Music、TikTok、カスタムなど、プラットフォームを想定したターゲットを選べます。これらはあくまで実用的な出発点であり、絶対的なルールではありません。

## ラウドネス目標

ラウドネス目標は、処理後ファイルが目指す Integrated LUFS の値です。

| 目標値 | 主な用途 |
|--------|----------|
| `-14 LUFS` | 多くの音楽リリースで扱いやすい、ストリーミング想定の目標。 |
| `-16 LUFS` | Apple Music 系、ポッドキャスト、スピーチ、ヘッドルームを残したい素材向け。 |
| カスタム | 納品仕様が明確な場合や、別ラウドネスを試聴したい場合。 |

大きければよいというものではありません。目標値が高すぎてリミッターが働きすぎると、パンチを失ったうえで、配信側で結局下げられてしまうことがあります。

::: warning 目標値を上げても「大きいマスター」にはならない
配信プラットフォームは再生ラウドネスを正規化します。ある点を超えると、目標値を上げてもリミッターが余計に働くだけで再生時のラウドネスは増えず、パンチを失います。[ラウドネスマッチング](../concepts/loudness-matching.md) を参照してください。
:::

## True Peak シーリング

シーリングは、再構成やコーデック変換のあとに現れるピークからファイルを守るための上限です。ストリーミング向けの出発点としては `-1 dBTP` 前後がよく使われます。

コーデック変換の余裕を広く取りたい場合は `-2 dBTP` など低めに設定します。よりタイトなシーリングは、納品経路が明確で管理されている場合にのみ採用します。

## 戻すべきサイン

目標値を上げた直後にキック・スネア・子音が平らに潰れていく場合は、ラウドネスではなく密度のほうを上げすぎています。まずは目標を下げるか、前段のダイナミクスを緩めます。シーリングを上げて回避するとコーデック変換後のクリップリスクが増えるため、最初の解決策にはしないでください。

逆に、目標を下げてもミュージカルなインパクトがほとんど変わらないなら、余裕を残した設定のほうが安全です。配信プラットフォーム側で下げられる可能性が高いケースでは、少し低めの目標のほうが結果としてパンチを保てます。

## ターゲットの選び方

公開先から始め、続いて素材を聴きながら判断します。

1. もっとも近いプラットフォーム想定のプリセットを選ぶ。
2. 過剰な調整をせず、まず一度レンダリングする。
3. リミッターがグルーヴやトランジェントの印象を変えすぎていないか確認する。
4. マスターが小さく感じる場合、ラウドネスを上げる前に目標値を下げるか、ダイナミクスを緩める。
5. JSON レポートを書き出し、目標値・シーリング・チェーン構成を記録しておく。

:::: details 実装メモ

デモはプラットフォーム選択を LUFS 目標値に変換し、libsonare のラウドネスステージに渡しています。True Peak リミッターとラウドネスオプティマイザーは同じシーリング値を共有し、最終ゲインがピーク安全性を超えないようにしています。

書き出されるレポートには、選択中のプリセット・プラットフォーム・目標 LUFS・チューニング値・ソース指標・レンダリング後指標・実行ステージ名が記録されます。これにより、ブラウザでのレンダリング結果を CLI やアプリケーション組み込みの出発点として扱いやすくなります。

LUFS 目標に到達する前にシーリング制約に当たった場合、実装はピーク安全性を優先します。この状態では目標値を上げても出力ラウドネスが期待通り上がらないことがあります。レポートの適用ゲインと出力 LUFS を併読し、目標設定の問題なのか、前段のダイナミクス／低域コントロールの問題なのかを切り分けて判断します。

::::
